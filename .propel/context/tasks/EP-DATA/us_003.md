# US-003 — Immutable Audit Events Table with PostgreSQL Trigger Guard

## User Story

**As a** compliance officer,  
**I want** all platform events written to an append-only `audit_events` table protected by a PostgreSQL trigger that rejects any UPDATE or DELETE,  
**So that** the audit trail is tamper-proof and satisfies GDPR and ISMS compliance requirements.

## Business Value

An immutable audit trail is a regulatory and insurance requirement. Enforcing immutability at the database trigger level means no application code path — intentional or accidental — can alter historical records.

## Acceptance Criteria

### Scenario 1: UPDATE on audit_events raises an exception
**Given** the trigger is installed  
**When** any application code or direct SQL executes `UPDATE audit_events SET ...`  
**Then** the database raises `IMMUTABLE_AUDIT_RECORD` error and rolls back the transaction.

### Scenario 2: DELETE on audit_events raises an exception
**Given** the trigger is installed  
**When** `DELETE FROM audit_events WHERE ...` is executed even by the service account  
**Then** the database raises the error and no rows are deleted.

### Scenario 3: INSERT succeeds with correct schema
**Given** the audit service calls the event writer  
**When** a valid event record is inserted  
**Then** it is persisted with `id`, `actor_id`, `event_type`, `entity_type`, `entity_id`, `payload` (JSONB), `ip_address`, `user_agent`, and `created_at` auto-set.

### Scenario 4: Bulk event writes processed within 100 ms
**Given** 50 concurrent audit events are inserted  
**When** all INSERTs execute  
**Then** all complete within 100 ms with no deadlocks.
