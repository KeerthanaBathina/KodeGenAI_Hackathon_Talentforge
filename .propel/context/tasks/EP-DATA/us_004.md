# US-004 — Prisma Migration Framework with Seed Scripts and Rollback Procedures

## User Story

**As a** developer,  
**I want** a documented Prisma migration workflow with environment-specific seed scripts, a rollback runbook, and CI-enforced migration drift detection,  
**So that** schema changes are applied safely across environments and the database state always matches the deployed application version.

## Business Value

A disciplined migration workflow prevents schema drift between environments — the leading cause of staging-does-not-match-production bugs. Automated drift detection catches accidental schema changes before they cause incidents.

## Acceptance Criteria

### Scenario 1: CI fails if unapplied migrations exist
**Given** a migration file exists that has not been applied to the staging database  
**When** CI runs `prisma migrate diff`  
**Then** the pipeline fails with an explicit "database drift detected" message.

### Scenario 2: Seed script populates development environment
**Given** a fresh development database  
**When** `npx prisma db seed` runs  
**Then** at least one candidate, recruiter, HR user, requisition, and application exist for manual testing.

### Scenario 3: Rollback runbook executes without data loss
**Given** a migration is applied that adds a nullable column  
**When** the rollback procedure is followed  
**Then** the column is removed without affecting existing row data.

### Scenario 4: Production migration applied with zero downtime
**Given** a migration adds a new non-breaking index  
**When** `prisma migrate deploy` runs in production  
**Then** the migration completes without table locking and the health endpoint remains responsive throughout.
