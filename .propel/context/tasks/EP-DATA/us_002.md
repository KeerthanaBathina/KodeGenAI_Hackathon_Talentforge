# US-002 — Configuration and Policy Tables — Scoring Thresholds, Reason Codes, and Email Templates

## User Story

**As a** platform administrator,  
**I want** all configurable platform parameters (AI scoring thresholds, rejection reason codes, approval policies, email templates) stored in dedicated database tables with versioning,  
**So that** business rules can be updated at runtime without a code deployment.

## Business Value

Storing policy as data rather than code allows HR operations to adjust screening thresholds, add reason codes, and update email copy without engineering involvement. Versioning prevents breaking in-flight applications when policies change.

## Acceptance Criteria

### Scenario 1: New scoring threshold is active for applications submitted after effective date
**Given** admin updates the `ai_shortlist_threshold` from 0.70 to 0.75 with an `effective_from` of tomorrow  
**When** an application is submitted today  
**Then** the old threshold of 0.70 is used; applications submitted tomorrow use 0.75.

### Scenario 2: Reason code table populated with seed values
**Given** the seed script runs  
**When** `SELECT * FROM reason_codes` is executed  
**Then** at least 10 standard rejection reason codes exist and none can be deleted if referenced by a decision.

### Scenario 3: Email template renders all tokens
**Given** the `offer_extended` template exists with `{{candidate_name}}` and `{{role_title}}` tokens  
**When** the template renderer is called with matching data  
**Then** both tokens are replaced and no raw `{{...}}` strings appear in the output.

### Scenario 4: Approval policy config returns correct approver chain
**Given** a requisition with `compensation_band = "L5"`  
**When** the approval policy is queried  
**Then** the response includes the correct multi-tier approver list for that band.

## INVEST Checklist

- [x] Independent — configuration tables do not depend on feature epics
- [x] Negotiable — initial seed values and versioning strategy are negotiable

- [x] Testable — seed counts, effective-date logic, and token rendering are verifiable
