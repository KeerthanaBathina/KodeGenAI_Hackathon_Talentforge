# US-001 — Core Domain Schema — Candidates, Applications, Screenings, and Reviews

## User Story

**As a** developer,  
**I want** the complete core database schema (candidates, profiles, requisitions, applications, resumes, screenings, reviews, users) defined as Prisma models with all foreign keys, unique constraints, and query-optimised indexes,  
**So that** every feature epic has a validated, type-safe storage layer from Sprint 1.

## Business Value

A correct schema with enforced constraints prevents data corruption in production and eliminates expensive late-stage schema migrations. Prisma models provide compile-time type safety across the entire API surface.

## Acceptance Criteria

### Scenario 1: All 17 tables created with correct columns
**Given** the initial migration is applied  
**When** `prisma migrate status` is checked  
**Then** all 17 tables exist with correct column types, NOT NULL constraints, and default values as per the data model.

### Scenario 2: Foreign key integrity enforced
**Given** an `applications` record references a `requisitions` row  
**When** a DELETE is attempted on the referenced requisition while open applications exist  
**Then** the database raises a foreign key violation and rolls back.

### Scenario 3: Unique constraints prevent duplicates
**Given** `candidates` has a unique constraint on `email`  
**When** a second INSERT with the same email is attempted  
**Then** the database raises a unique constraint violation with an actionable error message.

### Scenario 4: Indexed queries perform within P95 target
**Given** 100 000 application rows exist  
**When** applications are filtered by `(requisition_id, status)`  
**Then** `EXPLAIN ANALYZE` shows an index scan and execution time is < 50 ms.

### Scenario 5: Row-Level Security restricts candidates to own data
**Given** RLS is enabled on `applications`  
**When** a JWT with `role = candidate` queries the table  
**Then** only rows where `candidate_id = auth.uid()` are returned.
