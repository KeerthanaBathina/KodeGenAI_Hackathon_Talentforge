# US-003 — Duplicate Application Prevention with Cooling Period Enforcement

## User Story

**As a** recruiter,  
**I want** the system to automatically prevent a candidate from applying to the same requisition twice and enforce a 90-day cooling period on rejected requisition families,  
**So that** the review queue is free of duplicate applications and the pipeline data is accurate.

## Business Value

Duplicate applications inflate pipeline metrics and create confusion in the HR review queue. Cooling periods give the team actionable SLA data and set clear re-application expectations for candidates.

## Acceptance Criteria

### Scenario 1: Duplicate application attempt blocked
**Given** a candidate already has an `active` or `submitted` application for a requisition  
**When** they attempt to apply to the same requisition again  
**Then** the "Apply" button is replaced by "Application In Progress" and the submission API returns HTTP 409 with a friendly message.

### Scenario 2: Cooling period prevents re-application after rejection
**Given** a candidate was rejected for "Software Engineer – Backend" 45 days ago  
**When** they view the same requisition  
**Then** the "Apply" button is disabled and a tooltip reads "Re-application available in 45 days."

### Scenario 3: Cooling period allows re-application after 90 days
**Given** a candidate was rejected 91 days ago  
**When** they view the requisition  
**Then** the "Apply" button is enabled and a banner reads "You previously applied — re-application allowed."

### Scenario 4: Duplicate check does not expose other candidates' status
**Given** the duplicate check API is called  
**When** the response is returned  
**Then** it only contains the requesting candidate's status; no other candidate application data is included.
