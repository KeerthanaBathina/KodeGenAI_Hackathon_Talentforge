# US-002 — HMAC Webhook Ingestion with Idempotent Score Processing

## User Story

**As a** platform engineer,  
**I want** assessment provider webhooks validated via HMAC-SHA256 signature and processed idempotently using the session token as deduplication key,  
**So that** score data is accepted only from authorised providers and duplicate webhook deliveries do not result in duplicate score records.

## Business Value

HMAC validation prevents forged webhook payloads from injecting fraudulent scores. Idempotent processing allows webhook retries from the provider (standard delivery-at-least-once behaviour) without corrupting the assessment record.

## Acceptance Criteria

### Scenario 1: Valid HMAC signature accepted and score stored
**Given** the provider sends a webhook with a valid `X-Signature-HMAC-SHA256` header  
**When** the endpoint validates the signature  
**Then** the payload is accepted, the score is written to `assessment_sessions.score`, and status is updated to `"completed"`.

### Scenario 2: Invalid HMAC signature rejected
**Given** a webhook arrives with an incorrect or missing signature header  
**When** the validation check runs  
**Then** the endpoint returns HTTP 401 and no data is written; a security event is logged to `audit_events`.

### Scenario 3: Duplicate webhook delivery is idempotent
**Given** the provider delivers the same webhook twice with the same session token  
**When** the second delivery arrives  
**Then** the endpoint returns HTTP 200 (acknowledging receipt) but no duplicate score record is created; the existing record is unchanged.

### Scenario 4: Score stored and application advanced to next stage
**Given** a valid score is ingested  
**When** processing completes  
**Then** the assessment session score is stored, the application's assessment stage is marked complete, and the next path stage becomes schedulable.
