# US-003 — Assessment Session Timer, Reconnect Handling, and Provider Configuration

## User Story

**As a** candidate,  
**I want** the assessment timer to persist across page reloads and reconnections, and be clearly visible throughout my test session,  
**So that** a network hiccup or accidental tab close does not cost me test time or result in an unfair session termination.

## Business Value

Persistent timers eliminate the most common assessment dispute — "the page reloaded and I lost time." Server-side timer state ensures the candidate cannot manipulate elapsed time by refreshing. Provider configuration management enables onboarding new assessment vendors without code changes.

## Acceptance Criteria

### Scenario 1: Timer persists across page reload
**Given** a candidate has 45 minutes remaining on their assessment  
**When** they reload the assessment page  
**Then** the timer resumes from the server-authoritative remaining time (not reset to the original duration).

### Scenario 2: Reconnect within 10 minutes resumes session
**Given** a candidate loses connectivity for 3 minutes  
**When** they reconnect  
**Then** the session resumes with the server-authoritative remaining time; elapsed offline time is counted against the total.

### Scenario 3: Session expires if candidate does not reconnect within 10 minutes
**Given** a candidate has been disconnected for 11 minutes  
**When** they attempt to reconnect  
**Then** the session is marked `"expired"` on the server, the test URL is deactivated, and the candidate sees "Your session has expired — please contact HR."

### Scenario 4: New assessment provider added without code deployment
**Given** an admin adds a new provider entry in the `assessment_providers` table with API endpoint, auth credentials, and HMAC secret  
**When** a recruiter selects that provider for a requisition  
**Then** assessments are launched and webhooks processed using the new provider's configuration.
