# US-001 — Comprehensive Audit Event Logging for All Platform Actions

## User Story

**As a** compliance officer,  
**I want** every authentication event, hiring decision, communication, and configuration change automatically written to the immutable audit log,  
**So that** a complete, tamper-proof record of all platform activity exists for regulatory audits and incident investigations.

## Business Value

A complete audit trail is a legal requirement for GDPR, employment law, and ISMS compliance. Immutable, comprehensive logging means the organisation can prove what happened, when, and by whom for any platform event.

## Acceptance Criteria

### Scenario 1: Authentication events are logged
**Given** a user logs in successfully  
**When** the authentication completes  
**Then** an `audit_events` record is created with `event_type = "auth.login"`, `actor_id`, `ip_address`, `user_agent`, and `created_at` within 100 ms.

### Scenario 2: Hiring decisions are logged with full context
**Given** an HR reviewer shortlists an application  
**When** the decision is saved  
**Then** an `audit_events` record captures `event_type = "decision.shortlist"`, `entity_type = "application"`, `entity_id`, `reason_code_id`, `actor_id`, and timestamp.

### Scenario 3: Configuration changes are logged
**Given** an admin updates the AI shortlist threshold  
**When** the save completes  
**Then** an `audit_events` record captures `event_type = "config.threshold_updated"`, old value, new value, `actor_id`, and timestamp.

### Scenario 4: Logging does not increase API latency by more than 10 ms
**Given** audit logging is enabled  
**When** a `POST /decisions` request is benchmarked with and without logging  
**Then** the difference in P95 latency is less than 10 ms (logging is fire-and-forget via BullMQ job).
