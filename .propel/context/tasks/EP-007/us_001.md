# US-001 — Prerequisite Validation Before Enabling Final Decision Controls

## User Story

**As a** hiring manager,  
**I want** the final hiring decision controls to be locked until all required interview stages and assessments are marked complete,  
**So that** no hiring decision can be made on an incomplete evaluation record.

## Business Value

Gating the decision step on completion prevents premature or legally-challengeable decisions. It ensures the full evaluation dataset exists before a consequential outcome is recorded.

## Acceptance Criteria

### Scenario 1: Decision panel locked when stages are incomplete
**Given** a candidate has completed 2 of 3 required interview stages  
**When** the hiring manager opens the final decision panel  
**Then** the "Make Decision" controls are greyed out and a checklist shows which stages are complete (green tick) and which are pending (grey circle).

### Scenario 2: Decision panel unlocked when all stages complete
**Given** all required stages are marked complete  
**When** the hiring manager opens the decision panel  
**Then** the "Make Decision" controls are active and the prerequisite checklist shows all ticks green.

### Scenario 3: API enforces prerequisite check server-side
**Given** a stage is incomplete  
**When** a POST to `/decisions` is sent directly (bypassing the UI)  
**Then** the API returns HTTP 422 "Prerequisite stage [name] is not complete" and no decision record is created.

### Scenario 4: Stage completion state is real-time
**Given** the hiring manager has the decision panel open  
**When** an interviewer submits the last pending scorecard  
**Then** the checklist item turns green via WebSocket push without requiring a page refresh.

## INVEST Checklist

- [x] Independent — prerequisite check standalone; does not alter any stage
- [x] Negotiable — required stages and prerequisite rules are negotiable
- [x] Valuable — prevents incomplete hiring decisions
- [x] Estimable — 3 SP; gate query + server-side validation
- [x] Small — validation gate only; decision submission is US-002
- [x] Testable — lock/unlock state and HTTP 422 are directly verifiable
