# US-002 — Policy and Threshold Editor with Versioning and Effective Date Isolation

## User Story

**As a** platform administrator,  
**I want** to create and update AI screening thresholds, approval policies, and hiring rules with effective dates, ensuring in-flight applications are not affected by policy changes,  
**So that** policy updates are safe to make at any time without disrupting ongoing hiring cycles.

## Business Value

Effective-date isolation prevents a policy change from retroactively altering in-progress applications — a critical fairness and compliance requirement. Versioning ensures every policy decision is auditable and reversible.

## Acceptance Criteria

### Scenario 1: New policy version with future effective date does not affect current applications
**Given** the admin sets a new AI shortlist threshold from 0.70 to 0.75 with an `effective_from` of next Monday  
**When** applications submitted this week are screened  
**Then** the 0.70 threshold is used; applications submitted from Monday onwards use 0.75.

### Scenario 2: Policy editor shows change history
**Given** the admin opens the shortlist threshold policy  
**When** the history tab is selected  
**Then** a table shows all previous versions with effective date, changed-by user, and old vs. new value.

### Scenario 3: Invalid policy value rejected
**Given** an admin enters a threshold value of 1.5 (out of 0–1 range)  
**When** they attempt to save  
**Then** inline validation shows "Threshold must be between 0 and 1" and the save is blocked.

### Scenario 4: Approval policy change applies to new offer decisions only
**Given** an approval policy is updated to add a new approver tier  
**When** existing pending-approval decisions are in the queue  
**Then** those decisions continue through the original approval chain; only decisions submitted after the effective date use the new chain.

- [ ] Validation rules enforced server-side; range errors surfaced to UI
