# US-003 — Configure Supabase PostgreSQL with Prisma ORM and Upstash Redis

## User Story

**As a** developer,  
**I want** Supabase PostgreSQL connected via Prisma ORM with a tuned connection pool and Upstash Redis available for caching and rate limiting,  
**So that** every application service has a type-safe, production-grade data layer from the start of feature development.

## Business Value

Type-safe database access via Prisma eliminates entire classes of runtime errors. Managed Supabase + serverless Upstash Redis remove infrastructure operational overhead and scale automatically.

## Acceptance Criteria

### Scenario 1: Prisma client connects on startup
**Given** `DATABASE_URL` points to Supabase  
**When** the backend starts  
**Then** Prisma establishes a connection pool (min 2, max 10) and a test query `SELECT 1` returns within 100 ms.

### Scenario 2: Redis PING succeeds on startup
**Given** `UPSTASH_REDIS_REST_URL` and token are configured  
**When** the health endpoint is called  
**Then** `redis: "ok"` is in the response and round-trip latency is < 50 ms.

### Scenario 3: Rate limiting returns HTTP 429 at threshold
**Given** a public endpoint is called 101 times from the same IP within 60 seconds  
**When** the 101st request arrives  
**Then** the API responds HTTP 429 with a `Retry-After` header; authenticated users share a separate 1 000 req/min limit.

### Scenario 4: P95 query latency within target
**Given** 50 concurrent database requests  
**When** all queries execute simultaneously  
**Then** P95 latency stays below 500 ms and no connection timeout errors occur.
