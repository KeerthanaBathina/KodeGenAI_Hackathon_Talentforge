# US-002 — Deploy Node.js/Express Backend to Railway.app with Zero-Downtime Rolling Strategy

## User Story

**As a** developer,  
**I want** the Express.js API and Socket.IO WebSocket server deployed to Railway.app using a rolling deployment strategy with health-check gating,  
**So that** the API is always reachable during releases and any failed deployment automatically rolls back without manual intervention.

## Business Value

Zero-downtime deploys protect the candidate and HR experience during releases. Automatic rollback on failed health checks prevents broken code from reaching users.

## Acceptance Criteria

### Scenario 1: Successful deployment with health check
**Given** a new build is pushed  
**When** Railway deploys the new container  
**Then** it only cuts over after `GET /health` returns `{"status":"ok","db":"ok","redis":"ok"}` with HTTP 200; previous instance keeps serving until then.

### Scenario 2: Unhealthy start triggers rollback
**Given** a missing environment variable causes the server to crash on start  
**When** Railway runs the health check  
**Then** the new container is stopped, the previous version continues serving, and a deployment failure alert fires.

### Scenario 3: WebSocket connectivity verified
**Given** the backend is deployed  
**When** a Socket.IO client connects to the WebSocket endpoint  
**Then** the connection succeeds in < 1 second and a `connected` acknowledgement is received.

### Scenario 4: Staging and production environments isolated
**Given** separate Railway environments exist for staging and production  
**When** `main` is deployed  
**Then** only production environment variables are injected into the production service.
