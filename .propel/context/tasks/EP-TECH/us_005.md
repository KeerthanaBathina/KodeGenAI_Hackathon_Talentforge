# US-005 — Security Middleware, Structured Logging, and OpenTelemetry Observability

## User Story

**As a** platform engineer,  
**I want** security headers, request-level rate limiting, Pino structured logging with PII masking, and OpenTelemetry traces exported to Grafana Cloud configured on the backend,  
**So that** the application is hardened against OWASP Top 10 vectors from day one and every request is observable for debugging and SLA monitoring.

## Business Value

Infrastructure-level security controls prevent attacks without requiring per-feature security code. Distributed tracing via OpenTelemetry gives the team full request visibility across the API, database, and BullMQ workers without managing tracing infrastructure.

## Acceptance Criteria

### Scenario 1: Security headers present on all responses
**Given** any HTTP request reaches the API  
**When** the response is returned  
**Then** headers include `Strict-Transport-Security`, `X-Frame-Options: DENY`, `X-Content-Type-Options: nosniff`, `Content-Security-Policy`, and `Referrer-Policy: strict-origin`.

### Scenario 2: OpenTelemetry trace visible in Grafana within 30 s
**Given** `POST /api/applications` is called  
**When** the request completes  
**Then** a trace with HTTP, Prisma query, and BullMQ enqueue spans is visible in Grafana Cloud within 30 seconds.

### Scenario 3: PII redacted from every log line
**Given** a registration request containing `email` and `password` fields  
**When** Pino emits the request log  
**Then** `password` is absent and `email` is masked to `j***@example.com`; the log includes `requestId`, `userId`, `method`, `path`, `statusCode`, `durationMs`.

### Scenario 4: TLS enforced — HTTP redirects to HTTPS
**Given** a client sends an HTTP (non-TLS) request  
**When** the request reaches the API gateway  
**Then** the server responds HTTP 301 redirecting to the HTTPS URL.
