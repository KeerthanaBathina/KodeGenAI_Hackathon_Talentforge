# US-004 — GitHub Actions CI/CD Pipeline with Quality Gates

## User Story

**As a** developer,  
**I want** a GitHub Actions pipeline that enforces type-checking, linting, and unit tests on every pull request and triggers automatic deployments on merge to `main`,  
**So that** broken code never reaches production and the team receives sub-5-minute feedback on every change.

## Business Value

Automated quality gates reduce production defect rates and eliminate manual deploy steps. A fast feedback loop keeps developer flow uninterrupted.

## Acceptance Criteria

### Scenario 1: CI runs on every PR and completes in < 5 minutes
**Given** a pull request is opened  
**When** the CI workflow triggers  
**Then** TypeScript type-check, ESLint, unit tests, and production build all complete within 5 minutes.

### Scenario 2: Failing test blocks merge
**Given** a unit test is failing in the PR branch  
**When** CI runs  
**Then** the workflow exits non-zero, the PR merge button is disabled by branch protection, and the failing step is identified in the Actions log.

### Scenario 3: CD deploys automatically on merge to main
**Given** all CI checks pass and a PR is merged  
**When** the CD workflow triggers  
**Then** the frontend deploys to Vercel and backend to Railway.app without any manual action.

### Scenario 4: Secrets never appear in workflow logs
**Given** `DATABASE_URL` and `JWT_SECRET` are used in a workflow step  
**When** the workflow runs  
**Then** both values are masked (`***`) in every log line.

### Scenario 5: Dependency vulnerability scan fails on HIGH CVE
**Given** a dependency with a known HIGH-severity CVE is present  
**When** the Dependabot / npm audit scan runs  
**Then** the CI check is marked failed and a PR comment identifies the affected package.

## INVEST Checklist

- [x] Independent — pipeline config independent of application code
- [x] Negotiable — specific workflow steps and parallelism are negotiable
- [x] Valuable — quality gate for all future development work
- [x] Estimable — 5 SP; GitHub Actions is well-understood tooling
- [x] Small — pipeline definition only, no app code changes
- [x] Testable — pass/fail status is directly observable in GitHub
