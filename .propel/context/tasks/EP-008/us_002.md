# US-002 — Email Delivery via Resend API with Exponential Backoff Retry

## User Story

**As a** platform engineer,  
**I want** all platform emails sent via the Resend API through a BullMQ delivery queue with exponential backoff retry on failure,  
**So that** transient API outages do not result in lost notifications and persistent failures are surfaced to operations for intervention.

## Business Value

Queued, retriable email delivery prevents notification loss during Resend API outages. Exponential backoff reduces pressure on the provider during degraded-service events while maximising eventual delivery probability.

## Acceptance Criteria

### Scenario 1: Email delivered within 60 s for first-attempt success
**Given** an email job is enqueued  
**When** the Resend API call succeeds on the first attempt  
**Then** the email is delivered and the delivery status record shows `"sent"` with a timestamp within 60 seconds.

### Scenario 2: Retry on HTTP 500 with exponential backoff
**Given** the Resend API returns HTTP 500  
**When** the first delivery attempt fails  
**Then** the job retries at 30 s, 2 min, 8 min, 32 min, and 128 min (5 attempts total); each retry delay doubles the previous.

### Scenario 3: Permanent failure after 5 attempts moves to DLQ
**Given** all 5 delivery attempts fail  
**When** the last attempt exhausts  
**Then** the job moves to the dead-letter queue, the email status record shows `"failed"`, and an ops alert is fired.

### Scenario 4: Duplicate email prevention via idempotency key
**Given** the same notification event fires twice (e.g., a webhook retry)  
**When** the second email job is enqueued  
**Then** the Resend API is called with the same idempotency key, preventing a duplicate email delivery.
