# US-004 — Interview Lifecycle State Machine — No-Show, Reschedule, Cancel, and Automated Reminders

## User Story

**As a** recruiter,  
**I want** to record no-shows, initiate reschedules, and cancel interviews through a state machine that enforces valid transitions, and have automated 24-hour and 1-hour reminder emails sent to all participants,  
**So that** the interview record always reflects reality and participants always have timely reminders.

## Business Value

A state machine with enforced transitions prevents invalid status combinations (e.g., rescheduling a cancelled interview). Automated reminders reduce no-show rates by up to 30% without manual recruiter effort.

## Acceptance Criteria

### Scenario 1: 24-hour reminder sent automatically
**Given** an interview is scheduled for tomorrow at 10:00 AM  
**When** the current time is 10:00 AM today (24 hours before)  
**Then** reminder emails are sent to the candidate, all panelists, and the recruiter; the email uses the `interview_reminder_24h` template.

### Scenario 2: 1-hour reminder sent automatically
**Given** an interview is scheduled in 1 hour  
**When** the BullMQ scheduler job fires  
**Then** a second reminder email is sent using the `interview_reminder_1h` template.

### Scenario 3: No-show recorded with outcome
**Given** an interview time has passed without scorecard submission  
**When** the recruiter marks the interview as "No Show"  
**Then** the status transitions to `"no_show"`, the candidate's no-show count increments, and a decision is required (reschedule or reject application).

### Scenario 4: Reschedule creates new interview and archives old
**Given** the recruiter initiates a reschedule  
**When** a new time slot is confirmed  
**Then** the original interview status changes to `"rescheduled"`, a new interview record is created linked to the same application, and new calendar invites are sent.

### Scenario 5: Invalid state transition rejected
**Given** an interview has status `"completed"`  
**When** the API receives a request to mark it `"no_show"`  
**Then** the API returns HTTP 422 with "Cannot transition from completed to no_show" and no state change occurs.
