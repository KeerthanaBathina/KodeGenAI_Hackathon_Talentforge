# US-002 — OTP Resend Rate Limiting with Lockout Window

## User Story

**As a** security engineer,  
**I want** OTP resend requests limited to 3 per 15-minute window via Upstash Redis counters,  
**So that** the verification endpoint cannot be abused to flood candidates with emails or brute-force OTP values.

## Business Value

Unthrottled OTP resend endpoints are a common abuse vector for email flooding and OTP brute-force attacks. Redis-backed rate limiting with a lockout window stops both vectors without impacting legitimate users.

## Acceptance Criteria

### Scenario 1: Third resend within 15 minutes succeeds
**Given** a candidate has requested 2 resend OTPs in the last 15 minutes  
**When** they click "Resend" a third time  
**Then** a new OTP is sent and the counter increments to 3.

### Scenario 2: Fourth resend within 15 minutes returns lockout message
**Given** a candidate has already requested 3 resend OTPs in the last 15 minutes  
**When** they attempt a fourth resend  
**Then** the API returns HTTP 429 and the UI displays "Too many requests — please wait X minutes" with a countdown showing time until lockout expires.

### Scenario 3: Counter resets after 15-minute window
**Given** a candidate was locked out 15 minutes ago  
**When** they click "Resend" now  
**Then** the request succeeds, a new OTP is sent, and the counter resets to 1.

### Scenario 4: Rate limit key is per-email, not per-IP
**Given** two candidates with different emails resend from the same IP  
**When** both request resends  
**Then** each candidate's counter is independent and one candidate's lockout does not affect the other.

## INVEST Checklist

- [x] Independent — rate limiting layer independent of OTP logic
- [x] Negotiable — limit counts and window duration are negotiable
- [x] Valuable — security control required before registration goes live
- [x] Estimable — 3 SP; Redis counter pattern is standard
- [x] Small — single endpoint enhancement only
- [x] Testable — counter state and 429 response are directly verifiable
